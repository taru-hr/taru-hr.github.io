How the HR dashboard works — explained for absolute beginners

No coding experience? Perfect. This is the friendly, jargon-free tour of how the interactive HR dashboard is put together. By the end you'll understand every moving part — and see that none of them are magic.

By Taru Parttimaa · HR Specialist

If you've looked at the HR analytics dashboard — the one with the headcount bars, the turnover line and the filters you can click — you might assume it took a team of engineers and some expensive software. It didn't. It's built from three simple ingredients that every website in the world uses, plus a small, tidy list of numbers.

Let's meet the ingredients first, then build the dashboard up one layer at a time. I'll show tiny snippets of the real code, but you never have to write or memorise any of it — I'll translate each one into plain English.

1. A web page is just three layers

Think of any web page like a person:

🦴

HTML is the skeleton. It decides what things exist and where — "here is a box, here is a heading, here is a table." Structure, nothing more.

🎨

CSS is the skin and clothes. It decides how those things look — the dark background, the gold colour, rounded corners, spacing.

⚡

JavaScript is the muscles. It makes things move and react — when you pick a different year, JavaScript is what notices and updates the numbers.

That's the whole toolkit. HTML, CSS, JavaScript. The dashboard is just these three, arranged thoughtfully around a list of HR numbers.

2. It all starts with the data

Before any charts, there's a plain list of numbers — think of it as a small spreadsheet, but written as text. Here's a shortened piece of it: 

// how many people, hires and leavers each team had in 2025
'Engineering': { headcount: 132, hires: 36, exits: 19 },
'Sales':       { headcount: 88,  hires: 41, exits: 30 },
'Marketing':   { headcount: 46,  hires: 12, exits: 8  },

Read it like a table. Each line is one department. headcount is how many people work there, hires is how many joined, and exits is how many left. The curly brackets { } are just a way of saying "these numbers belong together, to this team."

In plain English

Everything on the dashboard is calculated from this little list. Change a number here and every card, chart and table that uses it updates. The data is the single source of truth.

3. The boxes on screen (HTML)

Each of those four cards at the top — Headcount, Turnover, Time to hire, Engagement — is just a labelled box. In HTML a box is written with <div>, which literally means "a division," or "a section." Simplified, one card looks like this: 

<div class="kpi-card">
  <p>Total headcount</p>
  <p>430</p>
</div>

Tags come in pairs: <p> opens a paragraph and </p> closes it (the slash means "close"). So this says: "a card, containing the label Total headcount and the number 430." That's all an HTML tag ever does — it wraps a piece of content and gives it a name.

4. Making it look good (CSS)

On its own, that card would be black text on a white page. CSS adds the style. It works by picking a thing and giving it looks. For example: 

/* style every box named "kpi-card" */
.kpi-card {
  background: #1f1f22;   /* dark grey */
  border-radius: 16px;   /* rounded corners */
  padding: 18px;         /* breathing room inside */
}

Read it as a sentence: "for the kpi-card, make the background dark grey, round the corners, and add space inside." 16px means 16 pixels — pixels are just the tiny dots your screen is made of, so it's a way of saying "this big." Every colour, gap and font on the dashboard is one of these small instructions.

5. Drawing charts with no special software

Here's the part people assume needs a fancy library. It doesn't. The charts are drawn with something called SVG, which is just a way of describing shapes with words. A rectangle, for instance, is written like this: 

<rect x="80" y="10" width="238" height="20" />

That means: "draw a rectangle 80 across, 10 down, 238 long and 20 tall." A bar chart is nothing more than a few of these rectangles stacked in rows. The clever bit is deciding how long each bar should be, and it's simple maths: 

bar length = ( this team's headcount ÷ the biggest headcount ) × chart width

So the biggest team fills the whole width, and a team half its size gets a bar half as long. The code does this once for each department and — there's your chart. The line chart works the same way (dots joined up), and the donut is a circle sliced into coloured wedges.

🔗

Connect the dots. If you've ever done a connect-the-dots drawing, you already understand SVG charts. The computer works out where each dot goes from the numbers, then draws the lines and shapes between them.

6. Making it react to you (JavaScript)

Now the fun part — why the dashboard responds when you use it. This is JavaScript's job. The idea is called an event: the page quietly listens, and when something happens (you pick a year, you click a bar), it reacts. 

// when the year dropdown changes, redraw everything
yearSelect.addEventListener('change', redraw);

Translated: addEventListener means "listen for," 'change' is the moment you choose a different year, and redraw is the instruction "do the whole drawing again." So the page hears your choice, goes back to the data from step 2, recalculates the numbers, and paints fresh bars. It feels instant because the list of numbers is tiny.

Clicking a bar to focus on one team works exactly the same way — a different event ("click"), the same "recalculate and redraw" response.

7. Where the KPI numbers come from

The big numbers at the top aren't stored anywhere — they're worked out on the spot from the data. Each one is a small, familiar HR formula:

  • Total headcount — add up every team's headcount.
  • Turnover rate — exits ÷ headcount × 100, i.e. leavers as a percentage of the team.
  • Average time to hire — the average across teams, weighted by team size.
  • Engagement score — the same kind of size-weighted average of each team's score.

These are the exact sums you'd do in a spreadsheet. The only difference is the page does them for you the instant you change a filter.

8. Try it yourself

A tiny experiment

Open assets/js/dashboard.js and find the line for 'Engineering' near the top. Change the headcount from 132 to, say, 200 and save. Refresh the dashboard: the Engineering bar grows, the total headcount jumps, and the turnover rate shifts — all from that one edit.

That's the whole point of keeping the data in one place. You're editing HR figures, not really "programming" at all.

To see your change, you'll want to preview the site. Because the site loads its pieces in the background, it needs to be opened through a small local web server rather than by double-clicking the file. If that sounds intimidating, it's one line in a terminal from the project folder: 

python3 -m http.server 8000

…then visit http://localhost:8000 in your browser. (On the live GitHub Pages site it just works, with no setup.)

So… is that really all?

Really. A short list of numbers, boxes made with HTML, looks added with CSS, shapes drawn with SVG, and JavaScript listening for your clicks. No paid tools, no database, no chart library — just those pieces working together. Every impressive dashboard you've seen is the same handful of ideas, scaled up.

You don't need to remember any of the code here. The valuable thing is the mental model: data in one place, three layers to show it, and events to make it react. Once you can picture that, no dashboard looks like magic again.
